# 出图工具增强方案（讨论版）

> 本文档整合当前讨论结论，供明日继续细化或落地。最后更新：讨论后整理。

---

## 一、需求概览

1. **界面补充图标**：在关键位置增加图标，提升识别度。
2. **单次最多 20 张图 + 相册式展示**：支持一次请求多张（1～20），返回后以相册网格展示，可做「一张张出现」的渐进效果。
3. **基于文档/知识库出图**：支持「上传文档」与「企业知识库」两种来源；由 LLM 根据文档正文生成可编辑的出图提示词，用户编辑后再出图。
4. **Agent 架构：两个 Skill**：写画图提示词（LLM）与画图本身（画图模型）拆成两个 Skill，各用各的模型。
5. **解析能力**：文档→正文用专门解析库（代码），不把原始文件直接丢给大模型；大模型只接收已解析的正文并生成提示词。
6. **双 API 配置**：界面与后端支持两套配置——「LLM API（写提示词）」与「出图 API（画图）」；单 API 用户两处填同一套即可，多 API 用户可分别配置。

---

## 二、界面与交互

### 2.1 图标

- **方案**：引入 Lucide React，在以下位置增加图标。
- **位置建议**：
  - 侧边栏「出图 API」：Settings / Key
  - 「描述画面」：PenLine / ImagePlus
  - 按钮「出图」：Sparkles / Image；「重试」：RefreshCw
  - 区域「已出图」：Images；空状态：ImageOff
  - 「选入 PPT」：CheckSquare；「导出为 PPT」/「下载 PPT」：Download、FileDown

### 2.2 多图与相册

- **出图张数**：创作区增加 1～20 的数字选择（默认 1），请求时传 `n`。
- **后端**：`/api/generate-image` 支持 `n`（1～20），返回 `images: string[]`（base64 数组）。
- **前端**：收到多张后依次追加到列表（可加短延迟做渐进展示）；「已出图」区域改为响应式网格（如 2～4 列），单卡为缩略图 + 选入 PPT 勾选 + prompt 摘要，整体呈相册感。

## 三、Agent 侧：两个 Skill

| Skill | 职责 | 使用模型 | 输入 → 输出 |
|-------|------|----------|-------------|
| **写画图提示词**（新） | 根据文档正文或用户描述，生成/润色出图用 prompt | 大语言模型（LLM） | 文档正文 / 用户描述 → `list[str]` 提示词 |
| **画图**（现有 image_gen_skill） | 根据单条 prompt 调用画图 API，得到图片；导出 PPT | 画图模型（DALL·E 等） | prompt → 图片 bytes / 多张图 |

- **画图模型**只做「按提示词出图」，不碰文档、不写 prompt。
- **大模型**只做「从文档/需求到提示词」，不直接调画图 API。
- 编排：先调「写提示词」Skill → 用户编辑 prompt 列表 → 再对每条调「画图」Skill。

**代码层面**：
- 现有 `image_gen_skill` 保持：只负责 prompt → 图、导出 PPT，内部用 `image_adapter`。
- 新增例如 `image_prompt_skill`：输入正文/描述 + 可选参数（如 max_prompts）；内部调 `llm_adapter`；输出 `list[str]` 提示词。

## 四、基于文档出图

### 4.1 解析能力（不丢给大模型）

- **解析**：用**文档解析库**从 PDF/Word/Txt 中抽取**纯文本**（或按页/段切分）。
  - 示例：PyMuPDF、pdfplumber（PDF），python-docx（Word），.txt 直接读。
  - 输出：干净正文，可做长度截断（如前 8k 字），再交给下游。
- **写提示词**：LLM 的输入是**已解析的正文**，不是原始文件；只根据这段正文生成若干条「适合直接出图的提示词」，并在 system prompt 中约束风格、条数等。

若对接企业知识库：先通过 API 拿到文档正文（或摘要），再把这段正文交给「写提示词」Skill，不在解析层混入大模型。

### 4.2 文档来源

- **上传文档**：用户选择 PDF/Word/Txt → 后端用解析库提取正文 → 调用「写提示词」Skill。
- **知识库**：抽象为「文档获取器」（如 doc_id + 配置 → 正文）；先实现上传，知识库接入口预留（如 WPS 365 等）。

### 4.3 流程

1. 文档 → **解析（代码）** → 正文
2. 正文 → **写提示词 Skill（LLM）** → 多条 prompt
3. 用户在前端编辑这些 prompt（可选）
4. 对每条 prompt 调用 **画图 Skill** → 结果进相册 → 可导出 PPT

## 五、双 API 配置

- **配置 1：LLM API（写提示词）**  
  - 用途：写提示词 Skill，调用 chat/completions。  
  - 字段：api_key、base_url、model。

- **配置 2：出图 API（画图）**  
  - 用途：画图 Skill，调用 images/generations。  
  - 字段：api_key、base_url、model。

- **单 API 用户**：两处填同一套 key 和 base_url；若代理对 chat 与 images 用同一 model 则模型名可相同，否则各填各的。
- **多 API 用户**：写提示词用一套，画图用另一套。
- 实现：写提示词逻辑只读 LLM 配置，画图逻辑只读出图配置；前端/Streamlit 均暴露两套配置区域，不做「统一基模」自动路由。

## 七、实施顺序建议

1. 图标 + 多图（n≤20）+ 相册网格 + 渐进展示（不依赖文档能力）。
2. 双 API 配置：前端与 API 支持 LLM / 出图两套配置。
3. 文档解析模块：上传 → 解析库 → 正文。
4. 写提示词 Skill（image_prompt_skill）+ POST /api/document-to-prompts（仅 upload 分支）+ 前端「根据文档」上传与可编辑提示词列表。
5. 知识库：文档获取抽象 + 具体实现（如 WPS 365）+ 前端知识库入口。

## 八、已落地（第一版）

- **API**：`POST /api/generate-image` 支持 `n`（1～20），响应 `images: string[]`，兼容 `image_base64`（首张）。
- **前端**：Lucide 图标、单次张数选择、相册网格、多图渐进追加；`npm install` 安装 `lucide-react`。
